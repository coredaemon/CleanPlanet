# CODEX BUILD PROMPT — сайт CleanPlanet (полная сборка за 1–2 прогона)

> Это единый промпт для Codex. Задача — собрать **весь сайт целиком** (каркас, дизайн-система, все страницы, реальные тексты, калькулятор, формы, юридический блок 152-ФЗ, медиа-галерея, SEO, аналитика-заглушки) за один, максимум два прогона. После сборки останутся только точечные правки. Работай последовательно, не задавай уточняющих вопросов по мелочам — при неоднозначности выбирай самый простой рабочий вариант в рамках правил ниже.

## 0. Роль и режим

Ты — инженер, собирающий production-каркас статического сайта клининга. Действуй строго в рамках уже готовой документации проекта. Не меняй стек, палитру, логотип и не выдумывай факты о компании.

## 2. Технический стек и старт

- **Astro** (последняя стабильная, 5.x) + **TypeScript** (strict) + **Tailwind CSS** + минимум клиентского JS.
- Node 20+. Пакетный менеджер — npm.
- Полностью статическая сборка (`output: 'static'`). Никакого SSR, БД, Node-сервера в production.

Инициализация: создать проект Astro в корне (`D:\Проекты\site CleanPlanet`), не удаляя существующие файлы документации, папки `01/`, `02/`, `brand/` и `Тексты для сайта/`. Настроить Tailwind, TypeScript strict, prettier. Добавить `@astrojs/sitemap`.

### 5.4. Трактовка маркеров в Markdown (ВАЖНО)
Тексты содержат служебные маркеры-подсказки для верстальщика. Они **не должны попадать на публичную страницу как обычный текст**. Обработать при рендеринге (remark-плагин или препроцессинг-скрипт):

1. **Редакторские заметки** — строки/блоки, целиком состоящие из меток вида `> [УТОЧНИТЬ ...]`, `[ДОБАВИТЬ ...]`, `[ПОЛНОЕ НАИМЕНОВАНИЕ ИП]` и т.п. → **удалять из публичного вывода** (можно оставлять как HTML-комментарий для команды). На сайте они не отображаются.
2. **Inline-плейсхолдеры** внутри предложений и ячеек таблиц (`[УТОЧНИТЬ]`, `[УТОЧНИТЬ ЦЕНУ]`) → заменять на нейтральное слово **«уточняется»** (или «по запросу» для цен). Реальные суммы не выдумывать.
3. **Блок `**Кнопки:**` + список** и **`**Кнопка:** X`** → это указание, какие CTA разместить. Сам блок-подсказку не выводить как текст; вместо него в макете ставится соответствующий компонент-кнопка (см. §7, §10). Стандартный набор hero-CTA: «Рассчитать стоимость», «Оставить заявку», «Отправить фотографии».
4. **Блок `**Поля формы:**` + список** → указание состава финальной формы заявки на странице. Не выводить как текст; на его месте — компонент формы `<LeadForm>` с перечисленными полями (см. §10).

Итог: на публичной странице — чистая проза (описания, чек-листы работ, таблицы видов уборки, FAQ, «как заказать», SEO-блок) + стандартные интерактивные компоненты в предусмотренных слотах. Никаких `**Кнопки:**`, `[УТОЧНИТЬ]`, редакторских ремарок в готовом HTML.

## 9. Калькулятор

- Логика **отдельно от интерфейса**: `src/data/prices.ts` (коэффициенты/тарифы) + `src/lib/calculator.ts` (чистые функции расчёта). Цены не разбрасывать по компонентам.
- Параметры (§17 AGENTS.md): тип помещения, вид уборки, число комнат, площадь, число санузлов, доп. услуги, степень загрязнения, срочность, за МКАД.
- **Режим цен по флагу.** В `prices.ts` завести флаг `pricesConfirmed: boolean` (сейчас `false`).
  - `pricesConfirmed === false` (текущий режим): калькулятор собирает параметры, показывает **сводку выбранного** и результат вида «Предварительная стоимость рассчитывается индивидуально — оставьте номер, и мы рассчитаем точно» + переход к заявке/отправке фото. **Никаких выдуманных сумм в рублях.**
  - `pricesConfirmed === true` (после того, как заказчик пришлёт тарифы и их впишут в `prices.ts`): те же функции считают и показывают диапазон «от … ₽» с пометкой «Предварительная стоимость». Переключение — только флаг + заполнение коэффициентов, без переписывания UI.
- UI: крупные переключатели/карточки вместо мелких select, визуальный прогресс, мгновенное обновление результата, пометка «Предварительная стоимость». Учитывать `prefers-reduced-motion`.

## 10. Формы

Формы (§18 AGENTS.md): основная заявка, быстрый расчёт, обратный звонок, срочная уборка, заявка для организаций, отправка фотографий. Составы полей брать из блоков `**Поля формы:**` соответствующих текстов.

Требования ко всем формам:
- отдельные `<label>` (placeholder не заменяет label), поля ≥ 48px, крупные чекбоксы/радио;
- валидация обязательных полей на клиенте и на сервере;
- **согласие на обработку персональных данных** — чекбокс, **НЕ отмечен по умолчанию**, обязателен для отправки, с ссылками на `/personal-data/` и `/privacy/` (см. §11);
- honeypot + rate limiting + серверная проверка вместо CAPTCHA;
- состояние отправки (loading/успех/ошибка), понятная ошибка, **не терять введённые данные** при временной ошибке;
- сохранять UTM-метки (на время сессии, напр. sessionStorage) и передавать с заявкой; передавать название страницы и выбранную услугу;
- после успешной отправки — редирект/экран `/thank-you/` и фиксация цели аналитики;
- загрузка фото: ограничение типов/размера, безопасные случайные имена, запрет исполнения в каталоге загрузок.

**PHP-обработчики** (`public/api/`): `send-lead.php`, `upload-images.php`, `config.example.php`. На этом этапе — рабочая структура с серверной валидацией/honeypot/rate-limit, отправкой на e-mail и уведомлением в Telegram, но **секреты только в `config.php`** (не в гите; добавить `.gitignore`; в репозитории — `config.example.php`). Реальные токены/почту не вписывать — заглушки в примере. Фронтенд деградирует корректно, если API ещё не подключён (показывает нейтральное сообщение), сборка от этого не падает.

## 11. Соответствие 152-ФЗ (юридический блок)

Реализовать по образцу типового сайта с обработкой ПДн (как в проекте юрфирмы):

1. **Согласие на обработку ПДн** во всех формах — чекбокс, не отмечен по умолчанию, обязателен, с текстом-ссылкой на `/personal-data/` и `/privacy/`. Отправка невозможна без явной отметки.
2. **Страницы:**
   - `/privacy/` — Политика конфиденциальности;
   - `/personal-data/` — Политика/согласие на обработку персональных данных;
   - `/terms/` — Пользовательское соглашение.
   Тексты — юридические заготовки с плейсхолдерами под ИП: `[ПОЛНОЕ НАИМЕНОВАНИЕ ИП]`, `[ИНН]`, `[ОГРНИП]`, `[АДРЕС РЕГИСТРАЦИИ]`, `[EMAIL]`. Всё в одном месте — `src/data/contacts.ts` (§16), чтобы заполнить позже одним касанием.
