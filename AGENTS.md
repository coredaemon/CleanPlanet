# Спецификация сайта клининговых услуг

## 1. Общая задача

Необходимо разработать современный многостраничный сайт клининговых услуг для Москвы и Московской области.

Основное назначение сайта:

* получение заявок из Яндекс.Директа;
* представление услуг частным лицам и организациям;
* предварительный расчёт стоимости уборки;
* приём заявок через формы, телефон и мессенджеры;
* создание технической основы для дальнейшего SEO-продвижения.

Сайт не является агрегатором, маркетплейсом или полноценной системой управления исполнителями.

На первом этапе не требуются:

* личные кабинеты;
* регистрация пользователей;
* автоматическое распределение заказов;
* CRM;
* сложная серверная логика;
* собственный AI-чат;
* онлайн-оплата.

Архитектура должна позволять позднее добавить онлайн-оплату, новые калькуляторы и дополнительные формы без полной переработки проекта.

---

# 2. Визуальный ориентир

В качестве общего ориентира по структуре и подаче используется сайт:

https://чистоздесь.рф/

Ориентир касается:

* крупного первого экрана;
* заметного призыва к расчёту стоимости;
* карточек услуг;
* отображения цен;
* описания состава уборки;
* последовательного расположения коммерческих блоков;
* повторяющихся форм заявки;
* ориентации сайта на получение обращения.

Сайт-ориентир нельзя копировать один в один.

Запрещается:

* копировать его тексты;
* повторять его фирменные цвета;
* воспроизводить точную композицию страниц;
* копировать графику, фотографии и иконки;
* создавать визуально неотличимый клон.

Необходимо сделать самостоятельный, более лёгкий, современный и аккуратный дизайн.

# 3. Характер дизайна

Сайт должен выглядеть:

* современно;
* технологично;
* чисто;
* спокойно;
* аккуратно;
* дружелюбно;
* достаточно дорого, но без премиального пафоса;
* профессионально, но не чрезмерно официально.

Визуальная ассоциация:

> современный городской сервис, через который легко заказать уборку.

Сайт не должен выглядеть:

* как шаблон WordPress;
* как сайт начала 2010-х годов;
* как типовой сайт небольшой клининговой фирмы;
* как доска объявлений;
* как детский или мультяшный проект;
* как корпоративный портал;
* как административная панель;
* как приложение с чрезмерным количеством карточек;
* как копия Qlean или другого крупного сервиса.

# 4. Цветовая система

У заказчика имеется логотип в белых, голубых и синих цветах. Сам файл логотипа будет добавлен позднее.

До получения логотипа использовать временную цветовую систему.

## Основные цвета

```css
--color-primary: #1769AA;
--color-primary-dark: #0B3E75;
--color-primary-light: #4DA3E3;

--color-accent: #55B8E8;
--color-accent-soft: #E8F5FC;

--color-background: #FFFFFF;
--color-background-alt: #F5F9FC;
--color-surface: #FFFFFF;

--color-text: #152536;
--color-text-muted: #607487;
--color-border: #DCE7EF;

--color-success: #218A62;
--color-error: #C94A4A;
```

Допускается небольшая корректировка оттенков для улучшения контраста и визуальной целостности.

После получения логотипа цвета необходимо адаптировать под него.

## Принципы использования цветов

Белый цвет должен оставаться основным фоном.

Синий используется для:

* основных кнопок;
* важных заголовков;
* ссылок;
* активных элементов;
* выделения цены;
* ключевых элементов калькулятора.

Светло-голубой используется для:

* фоновых секций;
* карточек;
* декоративных элементов;
* мягких градиентов;
* иллюстративных пятен.

Нельзя делать весь сайт насыщенно-синим.

Нельзя использовать кислотные оттенки.

Не использовать зелёную цветовую систему сайта-ориентира.

Розовые и сиреневые оттенки не использовать без отдельного согласования.

# 5. Типографика

Использовать современный гротескный шрифт с хорошей поддержкой кириллицы.

Предпочтительные варианты:

* Manrope;
* Inter;
* Onest;
* Golos Text;
* PT Root UI, если доступен законный способ подключения.

Не подключать более двух семейств шрифтов.

Предпочтительно использовать одно семейство с несколькими начертаниями.

## Заголовки

Заголовки должны быть:

* крупными;
* короткими;
* хорошо читаемыми;
* без чрезмерно жирного начертания;
* с умеренно плотным межстрочным интервалом.

На первом экране допустим заголовок размером от 42 до 64 px на десктопе.

На мобильных устройствах — от 30 до 40 px.

## Основной текст

Основной текст:

* от 16 до 18 px;
* с комфортным межстрочным интервалом;
* без длинных строк;
* с ограниченной шириной текстовых колонок.

# 6. Композиция и интерфейс

Использовать:

* большие отступы между секциями;
* просторную сетку;
* крупные кнопки;
* заметные цены;
* аккуратные скругления;
* лёгкие тени;
* тонкие границы;
* мягкие фоновые переходы;
* простые современные иконки;
* умеренную асимметрию.

Карточки не должны использоваться повсюду без необходимости.

Каждая секция должна иметь ясную визуальную функцию.

Необходимо избегать ощущения, что сайт собран из десятков одинаковых белых прямоугольников.

## Скругления

Рекомендуемые значения:

```css
--radius-small: 10px;
--radius-medium: 16px;
--radius-large: 24px;
--radius-pill: 999px;
```

## Тени

Использовать лёгкие, мягкие тени.

Запрещены:

* тяжёлые серые тени;
* чёрные контуры;
* неоморфизм;
* чрезмерное размытие.

# 7. Анимации

Анимации должны быть спокойными и функциональными.

Допустимы:

* плавное появление элементов;
* небольшое движение при наведении;
* плавное раскрытие FAQ;
* анимация изменения стоимости;
* смена активных табов;
* мягкая реакция кнопок;
* анимация прогресса квиза.

Запрещены:

* постоянное движение декоративных объектов;
* сложные параллакс-эффекты;
* видеофон;
* чрезмерные анимации при прокрутке;
* анимации, замедляющие взаимодействие;
* тяжёлые библиотеки ради нескольких эффектов.

Учитывать настройку `prefers-reduced-motion`.

Сайт преимущественно статический.

Для отправки заявок допускаются небольшие PHP-обработчики.

Примерная структура:

```text
public/
└── api/
    ├── send-lead.php
    ├── upload-images.php
    └── config.example.php
```

PHP используется только для:

* приёма заявок;
* валидации формы;
* отправки данных на электронную почту;
* отправки уведомления в Telegram;
* безопасной загрузки фотографий;
* ограничения спама.

Секретные данные нельзя размещать в клиентском JavaScript.

К секретным данным относятся:

* токены Telegram;
* пароли;
* SMTP-доступы;
* API-ключи;
* данные платёжной системы.

# 12. Работа с текстами

На первом этапе страницы создаются с временными текстами-заглушками.

Необходимо отделять содержание от визуальных компонентов.

Основные коммерческие тексты позднее будут подготовлены отдельно в Markdown-файлах.

Пример:

```text
src/content/services/
├── uborka-kvartir.md
├── generalnaya-uborka.md
├── uborka-posle-remonta.md
├── moyka-okon.md
├── uborka-ofisov.md
└── uborka-kommercheskih-pomeshcheniy.md
```

Статьи:

```text
src/content/articles/
├── chto-vhodit-v-generalnuyu-uborku.md
├── kak-rasschityvaetsya-stoimost.md
└── kak-podgotovit-kvartiru.md
```

Тексты нельзя жёстко зашивать внутрь компонентов.

Компоненты должны получать текст через props, данные или коллекции Astro Content.

До подготовки окончательных материалов нельзя:

* выдумывать историю компании;
* придумывать количество клиентов;
* придумывать отзывы;
* придумывать гарантии;
* придумывать опыт работы;
* указывать неподтверждённые цены;
* указывать несуществующий офис;
* писать о собственном штате сотрудников без подтверждения.

Для отсутствующих данных использовать нейтральные заглушки.

# 15. Первый экран

Первый экран должен сразу сообщать:

* какие услуги оказываются;
* где они оказываются;
* как узнать стоимость;
* как оставить заявку.

Предварительный вариант заголовка:

> Уборка квартир и помещений в Москве

Подзаголовок:

> Поддерживающая, генеральная и уборка после ремонта. Рассчитайте предварительную стоимость и оставьте заявку на удобное время.

Основные действия:

* «Рассчитать стоимость»;
* «Оставить заявку»;
* «Отправить фотографии».

На мобильной версии должны быть заметны:

* телефон;
* мессенджер;
* кнопка расчёта.

11. Повторная форма заявки.

Страницы должны подходить одновременно для:

* Яндекс.Директа;
* органической выдачи;
* переходов из внутренних разделов сайта.

# 17. Калькулятор

Калькулятор является предварительным.

Он не обязан показывать окончательную цену.

Допустимые результаты:

* цена «от»;
* примерный диапазон;
* предварительная стоимость;
* предложение отправить фотографии;
* предложение оставить телефон для уточнения.

Базовые параметры:

* тип помещения;
* вид уборки;
* количество комнат;
* площадь;
* количество санузлов;
* дополнительные услуги;
* степень загрязнения;
* срочность;
* удалённость за МКАД.

Логика расчёта должна храниться отдельно от интерфейса.

Пример:

```text
src/data/prices.ts
src/lib/calculator.ts
```

Цены не должны быть разбросаны по компонентам.

# 18. Формы

Необходимо предусмотреть следующие формы:

* основная заявка;
* быстрый расчёт;
* обратный звонок;
* срочная уборка;
* заявка для организаций;
* отправка фотографий.

Все формы должны:

* корректно работать на мобильных устройствах;
* проверять обязательные поля;
* иметь защиту от спама;
* показывать состояние отправки;
* показывать понятное сообщение об ошибке;
* не терять введённые данные при временной ошибке;
* сохранять UTM-метки;
* отправлять название страницы и выбранной услуги;
* после успешной отправки фиксировать цель аналитики.

Не использовать CAPTCHA, если можно обойтись honeypot, rate limiting и серверной проверкой.

---

# 19. Логотип

До получения настоящего логотипа использовать временный текстовый знак.

Пример:

```text
Название сервиса
Клининг в Москве
```

Не создавать постоянный логотип самостоятельно без отдельного задания.

Не привязывать архитектуру шапки к конкретным пропорциям временного знака.

После получения файла логотипа должно быть возможно быстро заменить временный вариант.

Поддерживаемые варианты:

* светлый логотип;
* тёмный логотип;
* компактный знак;
* полный вариант с названием.

---

# 20. Изображения

На первом этапе разрешается использовать временные стоковые изображения или нейтральные заглушки.

Необходимо предусмотреть последующую замену на:

* реальные фотографии выполненных работ;
* фотографии до и после;
* изображения квартир;
* изображения офисов;
* фотографии инвентаря;
* фотографии исполнителей.

Все изображения должны:

* иметь адаптивные размеры;
* использовать WebP или AVIF;
* иметь корректный `alt`;
* загружаться лениво, кроме изображения первого экрана;
* не ухудшать производительность.

Не использовать клишированные изображения людей, демонстративно держащих швабру и улыбающихся в камеру, если есть более естественная альтернатива.

# 21. Адаптивность

Сайт в первую очередь должен быть удобен на мобильном телефоне.

Обязательные контрольные ширины:

* 360 px;
* 390 px;
* 768 px;
* 1024 px;
* 1440 px;
* 1920 px.

Не должно быть:

* горизонтальной прокрутки;
* слишком мелких кнопок;
* неразборчивого текста;
* элементов, доступных только через hover;
* таблиц, выходящих за пределы экрана;
* форм с неудобными полями;
* перекрытия контента фиксированными элементами.

На мобильной версии предусмотреть нижнюю фиксированную панель:

* позвонить;
* написать;
* рассчитать.

# 22. SEO

Сразу заложить:

* уникальный `title`;
* уникальный `description`;
* один `H1`;
* логичную структуру `H2–H3`;
* человекопонятные URL;
* канонические ссылки;
* Open Graph;
* `robots.txt`;
* `sitemap.xml`;
* хлебные крошки;
* внутренние ссылки;
* страницу 404;
* изображения с `alt`;
* микроразметку;
* быструю загрузку;
* корректные редиректы.

Не создавать массово страницы под районы Москвы с одинаковым текстом.

Региональные страницы добавляются позднее только при наличии содержательного уникального материала.

# 26. Безопасность

Не хранить пользовательские данные без необходимости.

Формы должны иметь:

* серверную валидацию;
* ограничение частоты запросов;
* honeypot;
* проверку размера данных;
* фильтрацию файлов;
* безопасные случайные имена файлов;
* запрет исполнения файлов в каталоге загрузок.

Не выводить пользовательский ввод без экранирования.

Не хранить секретные ключи в Git.

Файл с секретами должен быть добавлен в `.gitignore`.

# 30. Главный принцип проекта

Сайт должен выглядеть как современный технологичный сервис, но оставаться технически простым.

Итоговая production-архитектура:

```text
Статические HTML, CSS и JavaScript
+
несколько небольших PHP-обработчиков форм
+
обычный виртуальный хостинг
```

Необходимо избегать лишней сложности.

Главный пользовательский сценарий:

> Выбрать услугу → узнать предварительную стоимость → оставить заявку → получить подтверждение.
