> Четыре независимые задачи (A–D). Правь по существу, БЕЗ слепых find-replace по всему проекту. После каждой правки следи, чтобы `npm run build` и `npm run lint` проходили без ошибок. Сохрани существующий дизайн, токены, компоненты, согласие 152-ФЗ и статический режим Astro. Тексты услуг лежат в `Тексты для сайта/*.md` (из них генерится `src/content/pages` скриптом `scripts/sync-content.mjs`) — правь источники.

## Задача A. Единая форма заявки (убрать дубли)

Сейчас на многих страницах ДВЕ формы: обычная заявка + отдельная «Отправьте фотографии для оценки». Это лишнее. Нужна **одна** форма везде, а фото — необязательное поле внутри неё.

### Что сделать
1. В `src/components/forms/LeadForm.astro` сделать так, чтобы **обычная форма всегда содержала необязательное поле загрузки фото** (несколько файлов, optional). Убрать отдельный режим-страницу «только фото» — оставить один универсальный вид формы. Вариант `formType="business"` (доп. поле «Организация») можно сохранить, но он тоже с опциональным фото и одной кнопкой.
2. Убрать вторые формы:
   - `src/pages/index.astro` — удалить секцию с `<LeadForm ... formType="photo" />` (строки с «Оценка стоимости по фотографиям»); оставить ОДНУ форму «Получить предварительный расчёт».
   - `src/layouts/ServiceLayout.astro` — удалить нижнюю секцию `<LeadForm ... formType="photo" />` («Оценка по фотографиям»); оставить ОДНУ форму (в сайдбаре).
   - `src/pages/projects.astro` — заменить `formType="photo"`-форму на обычную единую форму (заявка с опциональным фото).
   - Остальные страницы (`about`, `contacts`, `prices`, `reviews`, `services/index`, `kommercheskiy-klining`) уже с одной формой — просто убедиться, что это тот же единый компонент с опциональным фото.
3. Итог: на любой странице — **ровно одна** форма заявки. Никаких «сначала форма про фото, потом форма про заявку».

### Поля единой формы
Имя; Телефон (обязательно); Услуга (select); Площадь или адрес; Комментарий; **Фотографии (необязательно, можно несколько)**; чекбокс согласия ПДн (не отмечен по умолчанию, обязателен). Honeypot и UTM — сохранить.

## Задача B. Фото в форме — сделать надёжным (техническая часть)

Проблема: с телефонов фото бывают 20–30 МБ, почта их не примет. Решение — **сжимать изображения на клиенте перед отправкой**.

1. Перед отправкой формы в inline-скрипте `LeadForm`:
   - для каждого выбранного изображения нарисовать его на `<canvas>` с ограничением по большей стороне **≈1600px**, экспортировать в JPEG (`toBlob`, quality ≈0.8);
   - заменить оригинальные файлы в `FormData` на сжатые;
   - ограничить количество (например, до 5 фото) и итоговый размер; если фото не выбраны — форма работает как обычная заявка.
   - Небольшая подпись у поля: «Фото по желанию — сжимаются автоматически».
2. Серверная часть: единый обработчик (можно свести `send-lead.php` и `upload-images.php` к одному эндпоинту, принимающему `multipart/form-data`): валидация полей, honeypot, rate-limit, безопасные случайные имена, лимит размера/типа, отправка на e-mail + Telegram; при отсутствии файлов — обычная заявка. Секреты только в `config.php` (в гите — `config.example.php`).
3. Форма должна деградировать корректно, если API ещё не подключён (нейтральное сообщение), сборка от этого не падает.

> Если сжатие по каким-то причинам недоступно в браузере — отправлять как есть, но не более лимита; фото в любом случае необязательны.

## Задача D. Страница `/projects/` — миниатюры + лайтбокс

Сейчас фото выводятся огромными `<img>` во всю ширину. Нужно: **компактная сетка миниатюр**, по клику — **лайтбокс** с листанием.

### Данные
`src/data/projects.json` уже содержит `collages[]` (готовые коллажи «до/после», поле `src`, `thumb`, `alt`) и `pairs[]` (отдельные `before`/`after`, есть `thumb`). Сформировать единый массив элементов галереи двух типов:
- `type: 'pair'` — отдельные до/после (`before`, `after`);
- `type: 'collage'` — готовый коллаж (`src`).

### Сетка
- Плитка миниатюр (`thumb`), 2 колонки на мобиле, 3–4 на десктопе, единое соотношение сторон (`object-cover`, напр. 4/3), скругления/border как у остальных карточек, ленивые.
- Каждая миниатюра — кликабельная кнопка (доступная: `<button>` с `aria-label`).

### Лайтбокс (переиспользуй паттерн из `src/pages/reviews.astro`)
- Оверлей на весь экран, затемнение, кнопка закрытия, стрелки ‹ › для листания по ВСЕМ элементам, закрытие по Esc / клику по фону / стрелкам клавиатуры, блокировка прокрутки фона, счётчик «N из M».
- Для `type: 'pair'` — внутри лайтбокса показать **слайдер до/после с ползунком** (переиспользуй логику `BeforeAfterSlider.astro`: две картинки, `clip-path`, перетаскиваемый разделитель), большого размера (`max-h: 85vh`).
- Для `type: 'collage'` — показать полный коллаж `object-contain`, `max-h: 85vh`.
- Изображения в лайтбоксе — полноразмерные (`src`/`before`/`after`), в сетке — только `thumb`.

Итог: страница компактная, фото не растянуты на весь экран; по клику открывается крупный просмотр с листанием и сравнением до/после ползунком.

Сейчас название бренда почти не звучит. Добавить **естественные** упоминания «CleanPlanet» — НЕ вставлять механически в каждое предложение, не делать find-replace.

Где уместно (2–4 упоминания на страницу максимум, там, где читается natural):
- `Тексты для сайта/О сервисе клининговых услуг.md` — вводный абзац («Сервис CleanPlanet помогает организовать уборку…»), раздел «Наш подход», «Почему удобно обратиться через сайт», заключение. 3–5 упоминаний по тексту.
- Главная (`Клининговые услуги в Москве и Московской области.md`) — 1–2 упоминания в вводных абзацах.

## Ограничения (важно)

- **Никаких слепых массовых замен.** Каждую правку делать осмысленно, с учётом контекста (в прошлый раз слепая замена `[УТОЧНИТЬ]`→«Рассчитывается индивидуально» насажала бессмыслицы — не повторять).
- Не трогать: реквизиты/контакты в `src/data/contacts.ts`, дизайн-токены, компоненты каруселей/слайдера/строки доверия/«Почему мы», согласие 152-ФЗ, cookie-баннер.
- Не выдумывать цены, отзывы, факты. Цена по-прежнему «Рассчитывается индивидуально» / «от 5 000 ₽».
- Статический Astro, без новых тяжёлых зависимостей. Сжатие фото — на нативном canvas, без библиотек.

## Критерии приёмки

1. `npm run build` и `npm run lint` — без ошибок.
2. На каждой странице **ровно одна** форма заявки; в ней есть необязательное поле фото; отдельной «фото-формы» нет нигде.
3. Загруженные фото сжимаются на клиенте (проверить: большой файл → в отправке уменьшенный); без фото форма отправляется как обычная заявка.
4. `/projects/`: сетка миниатюр (не полноширинные `<img>`), по клику — лайтбокс с листанием; пары открываются слайдером до/после, коллажи — полным изображением.
5. Нет формулировок «Готовые коллажи», «Пары до и после», H1 «Примеры работ».
6. «CleanPlanet» естественно упоминается в «О сервисе», на главной и части страниц услуг; без шаблонного спама.
7. Согласие ПДн, cookie-баннер, строка доверия, «Почему мы», карусели, слайдер до/после — на месте и работают.
