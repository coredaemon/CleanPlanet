## Задача A. КОРНЕВОЙ БАГ: классы прозрачности вне шкалы Tailwind не генерируются

Проект на Tailwind **v3** — модификатор `/NN` работает только со значениями шкалы opacity (0,5,10,20,25,30,40,50,60,70,75,80,90,95,100). В коде есть значения вне шкалы — **эти классы молча не генерируются**, элементы остаются прозрачными. Подтверждено по собранному CSS.

Найти и заменить ТОЧЕЧНО (только эти суффиксы, не трогая валидные /50 /70 /75 /80 /90):

| Сейчас (битый) | Где | Заменить на |
|---|---|---|
| `bg-white/92` | `src/components/common/Header.astro` (липкая шапка) | `bg-white/95` (+ оставить `backdrop-blur`) |
| `bg-black/82` | `src/pages/projects.astro` (оверлей лайтбокса) | `bg-black/80` |
| `bg-white/96` | `src/components/common/MobileBottomBar.astro` | `bg-white` (панель должна быть непрозрачной) |
| `bg-white/12` | `src/pages/index.astro` (чипы «Уборка для бизнеса» на тёмном фоне) | `bg-white/10` |
| `border-white/12` | `src/components/common/Footer.astro` | `border-white/10` |
| `text-white/78` | Footer и др. | `text-white/80` |
| `text-white/65` | где встречается | `text-white/70` |

После замены проверь: `grep -rn "white/92\|white/96\|black/82\|white/12\|white/78\|white/65" src` — пусто, и в собранном CSS появились классы замен.

**Результат по шапке:** при прокрутке текст меню и телефон всегда читаемы (непрозрачный/95% белый фон + blur). Проверить на страницах с фото под шапкой.

## Задача B. Лайтбокс `/projects/` — починить полностью

Файл: `src/pages/projects.astro`. Симптомы с превью: фото распахивается почти на весь экран без тёмного фона, «не листается» на вторую, на мобильном каша (наезжает на cookie-баннер и нижнюю панель).

1. **Тёмный оверлей** — придёт из задачи A (`bg-black/80`). Лайтбокс должен перекрывать ВСЁ: шапку, cookie-баннер, мобильную нижнюю панель (у лайтбокса z-[100], у остальных ниже — проверить фактически).
2. **Размер изображения**: сейчас `#projects-media img { width: 100%; }` — портретные коллажи распахиваются во всю ширину. Заменить на `width: auto; max-width: min(92vw, 1100px); max-height: 80vh; margin-inline: auto; object-fit: contain;`. Слайдер-пара (`.project-ba`) — тоже ограничить: `max-width: min(92vw, 1100px)`, высота от aspect-ratio, но не более 80vh.
3. **Листание**: проверить и починить prev/next (стрелки должны листать по кругу через все 17 элементов; счётчик «N из 17» обновляется). Добавить листание клавиатурой ←/→ и Esc для закрытия (если нет). Добавить **свайп влево/вправо на тач-устройствах** (touchstart/touchend по горизонтали >40px — след./пред.).
4. **Мобильный вид**: оверлей на весь экран, изображение по центру, стрелки не перекрывают фото критично (можно сместить вниз под фото на узких экранах), крестик в правом верхнем углу доступен. Нижняя панель и cookie-баннер не должны быть видны поверх лайтбокса.
5. Клик по фону закрывает (есть), скролл-лок body (есть) — сохранить.

Тот же паттерн стрелок/свайпа применить к лайтбоксу отзывов `src/pages/reviews.astro`, если там те же проблемы.

## Задача C. ЕДИНАЯ форма заявки + попап-механизм (заменяет все формы на сайте)

### C1. Поля формы (`src/components/forms/LeadForm.astro` — переработать)

Ровно этот состав, без вариантов `formType` (варианты business/photo удалить; скрытые поля page/service/utm и honeypot — сохранить):

1. **Имя** — обязательное, label «Имя».
2. **Телефон** — обязательное, **маска ввода** `+7 (___) ___-__-__`: форматирование на лету при вводе, только цифры, максимум 10 цифр после +7 (сейчас можно вбить сколько угодно — это баг). Валидация перед отправкой: ровно 10 цифр, иначе не отправлять и показать подсказку.
3. **Услуга** — `<select>` с ПОЛНЫМ списком услуг из `src/data/services.ts` (все 15, shortTitle) + первый пункт «Выберите услугу» + последний «Другое». Поддержать предзаполнение (см. C2).
4. **Где убираться** — переключатель (радио-кнопки в стиле калькулятора): **«Москва»** (по умолчанию) / **«За МКАД»**. При выборе «За МКАД» появляется поле «Примерное расстояние от МКАД, км» (число, необязательное).
   Поле «Площадь или адрес» — УДАЛИТЬ (люди не хотят давать адрес).
5. **Чекбокс «Нужна срочная уборка»** (не отмечен по умолчанию).
6. **Фотографии** — label ровно: **«Фотографии (по желанию, до 5 изображений)»**. Убрать все подписи про форматы файлов и автоматическое сжатие. Само клиентское сжатие в JS — ОСТАВИТЬ (молча), лимит 5 файлов — оставить/добавить.
7. **Комментарий** — textarea, как сейчас.
8. **Согласие на обработку ПДн** — как сейчас (обязательное, НЕ отмечено, ссылки на /personal-data/ и /privacy/).

`public/api/send-lead.php`: принимать новые поля (location, mkadDistance, urgent) и включать их в письмо/Telegram. Поле «Организация» больше не приходит — убрать из обработки, если упоминалось.

### C2. Попап (модалка) с этой формой

- Один компонент-модалка (например `src/components/forms/LeadFormModal.astro`), подключается в `BaseLayout` — доступен на каждой странице.
- Открывается кликом по ЛЮБОМУ элементу с атрибутом `data-open-lead`. Опциональный `data-service="Название"` предзаполняет select услуги.
- Поведение: затемнение фона (bg-black/80), форма по центру, на мобильном — почти во весь экран (max-h ~92vh, внутренний скролл), крестик, Esc и клик по фону закрывают, скролл-лок body, фокус в поле «Имя» при открытии, `role="dialog" aria-modal="true"`.
- **Все кнопки-CTA выше подвала страницы открывают попап вместо прыжков по якорям.** Заменить `href="/#calculator"`, `href="#lead-form"`, `href="/services/…#lead-form"` на `<button data-open-lead>`:
  - Шапка — кнопка «Рассчитать» (название оставить) → попап.
  - Мобильная нижняя панель — «Рассчитать» → попап.
  - Hero — см. задачу D.
  - Карточки услуг (`ServiceCards.astro`) — кнопка «Заказать» → попап с `data-service` этой услуги. «Подробнее» остаётся ссылкой на страницу услуги.
  - Кнопки в сайдбаре калькулятора («Оставить заявку», «Отправить фотографии») — заменить на ОДНУ кнопку «Оставить заявку» → попап. Кнопку «Отправить фотографии» удалить везде по сайту.
- Встроенная форма в нижней части страницы — остаётся обычной секцией (не попап). В `dist` не должно остаться ни одного `#lead-form`/`/#calculator` в href.

## Задача E. Дедупликация секций расчёта/заявки — по каждой странице

Правило: **на странице ровно одна секция приёма заявки**. Калькулятор и форма не живут отдельными дублирующими секциями.

Сделать объединённую секцию «Расчёт и заявка» (правка `Calculator.astro`): слева — параметры калькулятора (как сейчас), справа в сайдбаре — итог («Предварительная стоимость от X ₽» + сводка) и СРАЗУ ПОД итогом встроенная единая форма (компактно) ЛИБО одна кнопка «Оставить заявку» → попап (выбери что аккуратнее по вёрстке; форма прямо в сайдбаре предпочтительнее — «всё в одной секции»).

Раскладка по страницам:

| Страница | Что оставить | Что убрать |
|---|---|---|
| `/` главная | Hero (1 кнопка) → TrustBadges → ServiceCards → **объединённая секция калькулятор+заявка** → Дом/Бизнес → InfoSections → BeforeAfter → Advantages → Faq | Нижнюю отдельную секцию `LeadForm «Получить предварительный расчёт»` — удалить (заявка уже в объединённой секции) |
| `/services/` каталог | Hero → контент → ServiceCards ×2 → **объединённая секция калькулятор+заявка** | Отдельные `<Calculator/>` и `<LeadForm «Помочь выбрать услугу»>` как две секции; плюс из ИСХОДНИКА `Тексты для сайта/Клининговые услуги в Москве.md` удалить целиком раздел `## Предварительный калькулятор` (строка ~474, текст-заглушка с «[РАСЧЕТНЫЙ ДИАПАЗОН]» — не рабочий калькулятор, мусор на странице) |
| Страницы услуг (`ServiceLayout`) | Hero (1 кнопка) → контент + сайдбар (цена + единая форма) → related-карточки | Секцию `<Calculator/>` — удалить (дубль: форма уже в сайдбаре) |
| `/prices/` | Прайс-таблица → **объединённая секция калькулятор+заявка** | Отдельную `LeadForm`-секцию |
| `/projects/`, `/reviews/`, `/about/`, `/contacts/`, `kommercheskiy-klining` | Одна встроенная единая форма внизу (как сейчас, но новая форма) | business-вариант формы (поле «Организация») |

## Ограничения

- Никаких слепых find-replace: каждую замену — с пониманием контекста.
- Не трогать: реквизиты `contacts.ts`, цены `prices.ts` и логику расчёта, слайдер до/после на главной, карусель отзывов, строку доверия, «Почему мы», юр-страницы.
- Дизайн — в текущих токенах/стиле; никаких новых библиотек (маска телефона и попап — ванильный JS).
- `sync-content.mjs`: если удаляешь раздел из исходника — регенерация произойдёт при build; следи, чтобы не осталось пустых секций.

1. `npm run build`, `npm run lint` — без ошибок.
2. `grep -rn "white/92|white/96|black/82|white/12|white/78|white/65" src -E` — пусто. Шапка при скролле непрозрачная и читаемая; мобильная нижняя панель непрозрачная.

7. В `dist` нет «Предварительный калькулятор» (текстовой секции) и нет «[РАСЧЕТНЫЙ ДИАПАЗОН]».
